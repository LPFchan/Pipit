## Product Contract

Pipit is the mobile companion app for the Immogen immobilizer ecosystem. It connects to the **Guillemot** deck receiver and acts as a phone-based companion or alternative to the **Uguisu** hardware key fob.

Pipit gives owners and trusted guests a secure phone key that can unlock or lock a compatible vehicle over BLE, manage provisioned slots, migrate or recover keys, and operate without a cloud service.

## Product Context

- Primary operator: an owner or trusted guest using a phone as a vehicle key.
- Primary vehicle environment: a Ninebot Max G30 or future compatible vehicle fitted with Immogen immobilizer hardware.
- Primary phone environment: iOS or Android with platform-native BLE, secure storage, camera, background-service, and foreground UI integrations.
- Primary problem being solved: replace or complement the dedicated Uguisu fob with a secure phone workflow for proximity unlock, manual lock or unlock, provisioning, migration, and recovery.
- Why this matters: the system must stay usable offline, preserve strict anti-replay guarantees, and tie administration to slot-based access control instead of cloud identity.

## Phone Key Model

The primary user-facing object is a provisioned phone key bound to one of the firmware's four slot identities. Each phone key is backed by a locally stored AES key and monotonic counter state.

Slot identities are fixed:

- internal slot `0`: Uguisu hardware fob
- internal slot `1`: owner phone
- internal slots `2` to `3`: guest phones

User-facing UI may label slots as 1 to 4 as long as the underlying internal-slot mapping remains unchanged.

## Key Lifecycle

1. A user provisions or recovers a phone key through a QR flow generated by Whimbrel or through a recovery path opened by the vehicle and firmware.
2. Pipit stores the imported slot key and counter state locally using platform secure storage.
3. In the background, Pipit watches for the vehicle's BLE advertisements and evaluates proximity thresholds to trigger lock or unlock behavior.
4. In the foreground, the user can manually lock or unlock from the fob-style home surface.
5. Onboarding and Settings use the BLE management transport to inspect slots, rename or revoke keys, provision guests, and perform recovery or migration flows.
6. The firmware and shared crypto layers enforce slot routing, strict monotonic counters, and authenticated management operations.

## User-Facing Capabilities

- **Proximity Unlock / Lock:** A low-power BLE service detects the vehicle and automatically unlocks on approach or locks on departure. Thresholding, hysteresis, and security rules use shared settings and BLE transport contracts rather than ad hoc UI logic.
- **Active Key Fob:** A focused foreground surface presents an interactive fob-style lock or unlock control. Manual commands use the same shared slot keys and strict counter discipline as background actions.
- **Multi-Slot Key Management:** Settings and onboarding support owner and guest provisioning, slot inspection, rename, revoke, migration, replacement, recovery, and local deletion. Management privileges must follow firmware slot semantics.
- **Offline Provisioning:** Owner setup uses Argon2id-encrypted QR payloads generated by Whimbrel. Guest provisioning can use plaintext QR payloads. Neither flow should require an external cloud service.
- **Android Maintenance Path:** Android may expose USB OTG maintenance tasks when they are isolated behind platform-specific backend contracts.

## Native App Boundary

- `shared/`: Kotlin Multiplatform shared core for AES-CCM payloads, Argon2id-backed provisioning helpers, QR parsing, onboarding gates, BLE transport contracts, counters, and shared settings keys.
- `androidApp/`: native Android app using Jetpack Compose, platform BLE services, foreground-service scanning, camera or onboarding integrations, SceneView-style fob presentation, and USB backend code.
- `iosApp/`: native iOS app using SwiftUI, platform BLE/CoreLocation behavior, secure storage, onboarding and settings surfaces, and the bundled viewer path for the fob presentation.
- `vendor/` and `assets/`: pinned or project-local viewer dependencies and fob/viewer assets that let Pipit ship without CDN-hosted runtime UI dependencies.

## Product Surfaces

- **Onboarding and recovery:** Import owner or guest keys, decrypt encrypted owner transfers, request required platform permissions, and recover keys when the vehicle or firmware has opened the appropriate path.
- **Home fob:** Present the primary manual lock or unlock control, foreground connection state, and fob-style visual feedback.
- **Settings and key management:** Inspect slots, tune proximity thresholds, manage owner or guest keys, and invoke supported maintenance or USB-assisted tasks without duplicating transport logic in UI code.

## Security And Protocol Invariants

- Monotonic counters remain strict with no replay window.
- Manual and proximity lock or unlock commands use AES-CCM authenticated payloads and the same slot-key discipline.
- Management sessions must preserve the existing PIN, SMP pairing, slot identification, provisioning-window, and management-permission model described in the architecture reference.
- Pipit's protocol, provisioning, QR parsing, and security behavior must stay compatible with Immogen firmware and the Whimbrel dashboard.
- A phone or UI state must not be treated as proof of management authority unless the firmware-recognized slot key and management flow allow that authority.

## Non-Goals

- A cloud-backed account system for key ownership or synchronization.
- A single cross-platform UI abstraction that hides platform-specific BLE or OS behavior.
- Relaxing slot permissions, pairing rules, or counter requirements for convenience.
- Treating the mobile app as the source of truth for firmware slot policy.

## Success Criteria

- Owners can provision and recover a phone key that interoperates with the current Immogen firmware and Whimbrel dashboard.
- Both mobile platforms can perform secure manual and proximity-based lock or unlock operations using the shared crypto model.
- Key management flows remain local-first, slot-aware, and testable without introducing cloud coupling or security regressions.
